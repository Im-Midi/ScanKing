# 避坑手册

都是这个项目里我真实踩过的坑，按"现象 → 原因 → 解法"记下来的，省得下次半夜再查一遍。动相关代码前建议扫一眼。

## PowerShell / Windows 脚本

**中文脚本乱码甚至解析失败**
现象：`.ps1` 输出"鏌ヨ"类乱码，某行报"无法识别 MB"。
原因：Windows PowerShell 5.1 把无 BOM 的 UTF-8 当 ANSI 读，中文字符串损坏后可能破坏引号配对。
解法：仓库内所有 `.ps1` 一律纯 ASCII（输出英文）；确需中文就存 UTF-8 **带 BOM**。

**cargo 一输出脚本就中止**
现象：`& cargo ... *>> log` 在第一行 Compiling 处抛 NativeCommandError。
原因：cargo 把进度写 stderr，PS 在 `$ErrorActionPreference="Stop"` 下把 stderr 行当异常。
解法：经 cmd 转发重定向：`& cmd /c "cargo ... >> log 2>&1"`。

**Start-Process 拿到的 ExitCode 是 null**
现象：命令明明成功，脚本判定失败。
原因：`-PassThru` 的进程对象若没先摸过 `.Handle`，退出后 `.ExitCode` 为 null。
解法：启动后立刻 `$null = $p.Handle`，结束后 `$p.WaitForExit()` 再读。

**管道死锁：子进程卡死不动**
现象：构建"假死"，无输出无退出。
原因：只实时读了 stdout，stderr 缓冲区（4KB）写满后子进程阻塞。
解法：stdout/stderr 都重定向到文件，轮询同步进日志（见 build_apk.ps1 的 Run 函数）。

**单元素数组 [0] 取出一个字符**
现象：JAVA_HOME 变成了 `C`。
原因：管道过滤剩单个字符串时结果是标量，`[0]` 取首字符。
解法：`@($x) | Select-Object -First 1`。

**Get-ChildItem -Include 把目录也算上**
现象：对 `.gradle` 目录 Get-Content 报拒绝访问。
原因：`*.gradle` 通配符匹配到了同名目录。
解法：加 `-File`。

## Rust / Cargo / 构建体积

**debug APK 400+ MB**
原因：debug 构建把全部调试符号打进 `.so`（tract/image 的 debuginfo 就几百 MB）。
解法：`[profile.dev] debug = 0`、`strip = "symbols"`；脚本里再用环境变量兜底：`CARGO_PROFILE_DEV_DEBUG=0`、`CARGO_PROFILE_DEV_STRIP=symbols`（环境变量优先于 Cargo.toml）。

**瘦身后 APK 还是 400 MB**
原因：Android 增量打包器在旧 APK 上原地更新，被删条目的数据残留为 zip"死空间"（目录里看不到）。
解法：打包前删除 `gen/android/app/build/outputs/apk` 强制全新打包（脚本已内置）。

**依赖要求超新 rustc**
现象：`kstring@2.0.3 requires rustc 1.96`。
解法：装最新 stable；或 `cargo update kstring@2.0.3 --precise 2.0.2` 钉旧版。

**debug 模式图像处理慢如蜗牛**
解法：`[profile.dev.package."*"] opt-level = 2`；tract 特殊——推理热点几乎全在 tract-linalg，只给它 opt 3，其余 tract crate O0，构建时间大幅下降而推理几乎不受影响。

## tract / ONNX / OCR

**加载 PaddleOCR 模型直接报错**
现象：`Failed to parse "p2o.DynamicDimension.0"`。
原因：paddle2onnx 导出的动态维度符号名带点号，tract 的符号解析器不认。
解法：加载前在内存里做**等长字节替换**（`p2o.DynamicDimension.` → `p2oxDynamicDimensionx`，长度相同不破坏 protobuf），再 `model_for_read`。

**动态形状**
tract 对每种输入形状要单独 optimize。det 固定 960×960 letterbox；rec 按宽度分桶（80/160/240/320/480/640）缓存 plan；cls 固定 48×192。

**大文件走 IPC 被截断**
现象：模型文件存在但残缺，加载失败。
解法：base64 分 2MB 块传输 + 追加写入 + 完成后校验字节数；`ready()` 用接近真实体积的阈值判断，加载失败自动删文件重装（自愈）。

## Tauri 2 Android 四连坑（JNI 分享/打开/相册）

按踩坑顺序，每一层都要过：

1. **opener 插件在安卓反序列化崩溃**（`Cannot construct instance of OpenArgs`）→ 放弃插件，自己用 ACTION_VIEW 实现打开。
2. **`ndk_context` 未初始化**（panic："android context was not initialized"，且 panic 会让前端永远等不到回应=转圈）→ 在 MainActivity.onCreate 里调自定义 JNI 方法 `nativeInit(applicationContext)`，Rust 侧存 JavaVM + Context 全局引用；所有原生调用包 `spawn_blocking + catch_unwind` 把 panic 变成可见错误。
3. **后台线程 FindClass 找不到 androidx 类**（ClassNotFoundException: FileProvider，但框架类都正常）→ 附着线程用系统类加载器，看不见 APK 里的类；在 nativeInit（主线程，类加载器正确）提前 find_class 并缓存全局引用。
4. **同类名 FileProvider 声明两个 = 静默失效**（URI 能生成，微信"获取资源失败"、PDF 查看器"文件不存在"）→ 运行时只有一个实例在响应；不要自己再注册，**复用 Tauri 自带 provider**（`${applicationId}.fileprovider`），给它的 `res/xml/file_paths.xml` 补 `<root-path>`。

另外：**EXTRA_STREAM 里的 URI 授权不会传递**（微信尤其敏感）→ Intent 必须同时 `setClipData(ClipData.newRawUri(...))` 再 addFlags(GRANT_READ)。

## 图像流水线

- **手机照片方向**：必须按 EXIF Orientation 转正后再处理，否则检测/矫正全歪。
- **增强滤镜的色差**：按通道做背景除法会把彩色证件"漂白"。正确做法：只在**亮度**上估计光照，同一增益乘到 RGB 三通道（色相饱和不变），增益限幅 [0.9, 1.8]。
- **大半径模糊的内存**：13MP 图的积分图要上百 MB。技巧：缩小到 ≤800px 做高斯，再按需双线性上采样（背景是低频的，质量无损失）。
- **矫正发糊**：双线性换 Catmull-Rom 双三次，文字边缘立竿见影。
- **拼页只缩不放**：放大必糊，宁可留白（缩放系数 `.min(1.0)`）。

## 其他

- Tauri 命令参数：Rust snake_case ↔ JS camelCase 自动转换，别在 JS 里写 doc_id。
- 前端资源打进 `.so`（tauri codegen），改 ui/ 后要重新构建才生效。
- APK 升级不清 App 数据 —— 模型/文档都在，但坏文件也在（所以要自愈逻辑）。
- WebView 的 `getUserMedia` 需要 CAMERA 运行时权限 + 清单声明，且个别 ROM 仍不放行，必须留系统相机/相册兜底入口。
